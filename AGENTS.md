# AGENTS.md

Architecture notes for the AI agents and ingestion pipeline in this Cloudflare
Worker. The worker archives web articles (Browser Rendering → D1 → Vectorize)
and exposes a NotebookLM-style chat experience on top of the archive.

## Runtime types

Bindings are typed by the global `Env` interface generated by `wrangler types`
into `worker-configuration.d.ts`. There is no hand-written bindings interface
and no `import` from `@cloudflare/workers-types` — that package is replaced by a
local empty stub (`tools/stub-workers-types`) via `pnpm.overrides`, because its
runtime types would otherwise conflict with the wrangler-generated ones. Re-run
`wrangler types` after editing `wrangler.jsonc`.

## AI Gateway

Every Workers AI call routes through a single AI Gateway. `src/backend/ai/gateway.ts`
is the only place the gateway is configured:

- `AI_GATEWAY_ID` — `"default-gateway"`.
- `AI_GATEWAY_OPTIONS` — passed as the third argument to `env.AI.run(model, input, opts)`.
- `getAI(env)` / `chatModel(env, model?)` — an AI SDK provider bound to `env.AI`
  and the gateway. Accepts `@cf/*` Workers AI models and `{provider}/{model}`
  strings, so any provider/model can be selected without code changes.
- `embed(env, text)` — gateway-routed embedding helper.

## Agents

Agents are Durable Objects exported from the worker entry point (`src/_worker.ts`)
and routed via `routeAgentRequest` from the `agents` SDK. Each agent is fully
modularized under `src/backend/ai/agents/<agentName>/`:

```
agents/<agentName>/
  index.ts        – the agent class (or public re-exports)
  types.ts        – agent-specific types
  methods/        – one file per method/tool group
```

### `NewsAgent` (`agents/newsAgent`)

Binding: `NEWS_AGENT`. An `AIChatAgent` that answers questions grounded in the
full archived corpus via Vectorize RAG (`searchArchive` tool).

### `ArticleChatAgent` (`agents/articleChat`)

Binding: `ARTICLE_CHAT_AGENT` (migration tag `v2-article-chat`). A
NotebookLM-style `AIChatAgent`; one Durable Object instance per chat session.
Session state (`pinnedArticleIds`, `sessionName`) is stored as agent state.

Tools:

- `search_sources` — RAG scoped to the pinned article sources.
- `set_session_sources` — pin article IDs as sources.
- `search_archive` — RAG across the whole archive.
- `spawn_artifact` — generate + deploy a PWA / mind map / summary card.
- `archive_article` — ingest URLs shared mid-conversation.

### `pwaSpawner` (`agents/pwaSpawner`)

Not a Durable Object — stateless methods used by `spawn_artifact`:

- `generatePWACode` — `env.AI.run` generates a single-file HTML artifact.
- `deployPWAToR2` — writes the HTML to the `SPAWNED_PWAS` R2 bucket.

## RAG

`src/backend/ai/rag/articleRag.ts` — Vectorize similarity search over the
archive. Query embeddings use the same model as ingestion so vectors are
comparable. `retrieveArticleContext` can be scoped to a set of article IDs.

## Ingestion

`src/backend/ai/ingest/ingestUrls.ts` — Browser Rendering → schema-enforced
metadata extraction (Workers AI JSON mode) → Vectorize embedding → D1. Article
inserts and property writes are batched (chunked) to avoid the per-row N+1
pattern. `extractUrls.ts` parses URLs out of free-form text (iOS share-sheet
dumps). Used by both the `POST /api/ingest` route (authenticated) and the
`archive_article` agent tool.

## D1 schema

`src/backend/db/schemas/` — one file per table, grouped by domain. `db/schema.ts`
re-exports them. `getDb(env)` (`db/index.ts`) returns the Drizzle client.

NotebookLM-specific tables (`schemas/articles/sessions/`):

- `chat_sessions` — articles pinned to a session.
- `spawned_artifacts` — every PWA / mind map / summary card generated, with its
  R2 key and worker-served URL.

## Frontend

- `/notebook` (`src/frontend/pages/notebook.astro` + `components/NotebookChat.tsx`)
  — three-pane NotebookLM UI: article sources, agent chat, spawned-artifact
  gallery. Mounted with `client:only="react"` (WebSocket hooks, no SSR).
- Spawned artifacts are served back through the worker at `/artifacts/*` from
  the `SPAWNED_PWAS` bucket; `GET /api/artifacts` lists them.
