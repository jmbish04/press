# AGENTS.md

Architecture notes for the AI agents and ingestion pipeline in this Cloudflare
Worker. The worker archives web articles (Browser Rendering ‚Üí D1 ‚Üí Vectorize)
and exposes a NotebookLM-style chat experience on top of the archive.

## Runtime types

Bindings are typed by the global `Env` interface generated by `wrangler types`
into `worker-configuration.d.ts`. There is no hand-written bindings interface
and no `import` from `@cloudflare/workers-types` ‚Äî that package is replaced by a
local empty stub (`tools/stub-workers-types`) via `pnpm.overrides`, because its
runtime types would otherwise conflict with the wrangler-generated ones. Re-run
`wrangler types` after editing `wrangler.jsonc`.

## AI Gateway

Every Workers AI call routes through a single AI Gateway. `src/backend/ai/gateway.ts`
is the only place the gateway is configured:

- `AI_GATEWAY_ID` ‚Äî `"default-gateway"`.
- `AI_GATEWAY_OPTIONS` ‚Äî passed as the third argument to `env.AI.run(model, input, opts)`.
- `getAI(env)` / `chatModel(env, model?)` ‚Äî an AI SDK provider bound to `env.AI`
  and the gateway. Accepts `@cf/*` Workers AI models and `{provider}/{model}`
  strings, so any provider/model can be selected without code changes.
- `embed(env, text)` ‚Äî gateway-routed embedding helper.

## Agents

Agents are Durable Objects exported from the worker entry point (`src/_worker.ts`)
and routed via `routeAgentRequest` from the `agents` SDK. Each agent is fully
modularized under `src/backend/ai/agents/<agentName>/`:

```
agents/<agentName>/
  index.ts        ‚Äì the agent class (or public re-exports)
  types.ts        ‚Äì agent-specific types
  methods/        ‚Äì one file per method/tool group
```

### `NewsAgent` (`agents/newsAgent`)

Binding: `NEWS_AGENT`. An `AIChatAgent` that answers questions grounded in the
full archived corpus via Vectorize RAG (`searchArchive` tool).

### `ArticleChatAgent` (`agents/articleChat`)

Binding: `ARTICLE_CHAT_AGENT` (migration tag `v2-article-chat`). A
NotebookLM-style `AIChatAgent`; one Durable Object instance per chat session.
Session state (`pinnedArticleIds`, `sessionName`) is stored as agent state.

Tools:

- `search_sources` ‚Äî RAG scoped to the pinned article sources.
- `set_session_sources` ‚Äî pin article IDs as sources.
- `search_archive` ‚Äî RAG across the whole archive.
- `spawn_artifact` ‚Äî generate + deploy a PWA / mind map / summary card.
- `archive_article` ‚Äî ingest URLs shared mid-conversation.

### `pwaSpawner` (`agents/pwaSpawner`)

Not a Durable Object ‚Äî stateless methods used by `spawn_artifact`:

- `generatePWACode` ‚Äî `env.AI.run` generates a single-file HTML artifact.
- `deployPWAToR2` ‚Äî writes the HTML to the `SPAWNED_PWAS` R2 bucket.

### `PressMcpAgent` (`agents/pressMcp`)

Binding: `PRESS_MCP` (migration tag `v4-press-mcp`). A remote MCP server built
with `McpAgent` from the Agents SDK. Served at `/mcp` with Streamable HTTP
transport (SSE fallback). External AI agents connect via `mcp-remote@latest`
or any MCP client that supports remote servers.

Tools:

- `submit_article` ‚Äî ingest URLs via Browser Rendering + AI extraction pipeline.
- `search_archive` ‚Äî Vectorize RAG similarity search across the corpus.
- `list_articles` ‚Äî browse archived articles with metadata and tags.
- `get_article` ‚Äî full article detail by ID (content, properties, tags).

Resources:

- `mcp://press/stats` ‚Äî archive statistics (total articles, last updated).

MCP client config example:

```json
{
  "mcpServers": {
    "press": {
      "command": "npx",
      "args": ["-y", "mcp-remote@latest", "https://<your-domain>/mcp"]
    }
  }
}
```

## REST API

The Hono OpenAPI application (`src/backend/api/index.ts`) serves:

- **`GET /openapi.json`** ‚Äî dynamically generated OpenAPI 3.1 spec.
- **`GET /swagger`** ‚Äî Swagger UI (reads `/openapi.json`).
- **`GET /scalar`** ‚Äî Scalar API reference (reads `/openapi.json`).
- **`POST /api/ingest`** ‚Äî legacy ingestion (API-key authenticated).
- **`POST /api/articles/submit`** ‚Äî full-featured article submission with OpenAPI
  docs. Accepts `{ urls: string[] }` or `{ urlsString: string }`. API-key
  authenticated via `WORKER_API_KEY` Secrets Store binding.

## RAG

`src/backend/ai/rag/articleRag.ts` ‚Äî Vectorize similarity search over the
archive. Query embeddings use the same model as ingestion so vectors are
comparable. `retrieveArticleContext` can be scoped to a set of article IDs.

## Ingestion

`src/backend/ai/ingest/ingestUrls.ts` ‚Äî Browser Rendering ‚Üí schema-enforced
metadata extraction (Workers AI JSON mode) ‚Üí Vectorize embedding ‚Üí D1. Article
inserts and property writes are batched (chunked) to avoid the per-row N+1
pattern. `extractUrls.ts` parses URLs out of free-form text (iOS share-sheet
dumps). Used by both the `POST /api/ingest` route (authenticated) and the
`archive_article` agent tool.

## D1 schema

`src/backend/db/schemas/` ‚Äî one file per table, grouped by domain. `db/schema.ts`
re-exports them. `getDb(env)` (`db/index.ts`) returns the Drizzle client.

NotebookLM-specific tables (`schemas/articles/sessions/`):

- `chat_sessions` ‚Äî articles pinned to a session.
- `spawned_artifacts` ‚Äî every PWA / mind map / summary card generated, with its
  R2 key and worker-served URL.

## Frontend

- `/notebook` (`src/frontend/pages/notebook.astro` + `components/NotebookChat.tsx`)
  ‚Äî three-pane NotebookLM UI: article sources, agent chat, spawned-artifact
  gallery. Mounted with `client:only="react"` (WebSocket hooks, no SSR).
- Spawned artifacts are served back through the worker at `/artifacts/*` from
  the `SPAWNED_PWAS` bucket; `GET /api/artifacts` lists them.
